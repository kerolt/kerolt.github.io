TC和XDP

tc 和 XDP 都能对网络数据包进行处理，但它们位于 不同的网络协议栈层次，因此性能、能力和适用场景差别都很大：

项目作用层级处理方向特点
XDP (eXpress Data Path)驱动层（最早）：网络设备驱动入口处（L2 之前）入站（Ingress）极快，绕过内核协议栈，可直接丢包/转发
tc (Traffic Control)内核网络栈层（L3/L4 之后）入站 + 出站通常用于带宽控制、QoS、流量整形等

下面这张图能清晰地看到 XDP 和 tc 各自处在什么位置：

                |              用户空间                  |

处理路径细节对比#

对比项XDPtc (ingress/egress)
所在层级NIC 驱动层（比内核协议栈还早）内核网络栈（L3/L4 层之间）
钩子位置驱动接收包 → DMA → XDPskb（socket buffer）进入或离开协议栈时
作用方向仅支持 ingress（入站）支持 ingress + egress
处理对象原始数据包（frame）封装后的 skb
性能极高（可在百万 pps 级别）中等（受内核调度和 qdisc 影响）
典型用途DDoS 防护、早期丢包、快速转发、内核旁路QoS、流量整形、限速、包分类
可编程性eBPF 程序（xdp 程序）eBPF 程序（cls_bpf）
返回动作XDP_PASS、XDP_DROP、XDP_TX、XDP_REDIRECTTC_ACT_OK、TC_ACT_SHOT、TC_ACT_REDIRECT
使用接口ip link set dev eth0 xdp obj prog.otc filter add dev eth0 ... bpf obj prog.o

数据处理流程举例#

XDP 流程（最早阶段）#

[网卡接收包]
   ↓
[XDP 程序执行]
   ├── XDP_DROP：直接丢包（不进入内核）
   ├── XDP_PASS：让包进入内核协议栈
   ├── XDP_TX：直接从驱动回发
   └── XDP_REDIRECT：转发到其他网卡或 AF_XDP socket

典型用途：

  • DDoS 防护
  • 内核旁路负载均衡（如 Cilium、Katran）
  • 零拷贝包处理（AF_XDP）

tc 流程（稍晚阶段）#

[包进入内核协议栈]
   ↓
[tc ingress qdisc 过滤器]
   ├── DROP（丢弃）
   ├── REDIRECT（重定向）
   ├── CLASSIFY（分类）
   ↓
[内核网络层路由与传输层处理]
   ↓
[tc egress qdisc 过滤器]
   ├── 限速、排队、优先级调度
   └── 最终发往网卡发送队列

典型用途：

  • QoS 优先级调度（HTB、HFSC）
  • 出口限速（TBF）
  • 网络延迟仿真（netem）
  • eBPF-based 监控/流量标记

现代系统（例如 Cilium、Facebook Katran、Cloudflare 内核加速）常常结合两者使用：

          +-----------+           +----------------+
 NIC ---> |    XDP    | 先级筛选 | tc ingress     |
          | (早丢包)  | -------> | (QoS/分类/标记)|
          +-----------+           +----------------+

例如：

  • 在 XDP 层 先快速丢弃明显恶意或无效流量；

  • 在 tc 层 做带宽管理、优先级调度。

性能层面比较#

项目XDPtc
执行位置驱动层内核网络栈
数据结构原始 bufferstruct sk_buff（开销较大）
延迟纳秒级微秒级
PPS 吞吐~10M pps~1M pps
零拷贝✅ 可实现 (AF_XDP)❌ 无法实现

总结：

XDP 是“在内核栈之前”的加速点，极端高性能；
tc 是“在内核栈内部”的控制点，更灵活但稍慢。
